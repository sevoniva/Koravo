# Security Notes

Koravo is currently optimized for local development and demonstration. The codebase already enforces several platform boundaries, but production deployments must harden authentication, authorization, network policy, and secret management.

## Current Boundaries

- All APIs use `/api/v1`.
- Controllers call application services and platform facades; controllers do not access Flowable native services directly.
- Tenant context comes from `X-Tenant-Id` in development.
- User context comes from `X-User-Id` in development.
- Request IDs are propagated through `X-Request-Id` or generated by the request filter.
- Key mutations write audit logs.
- Datasource secrets are encrypted and are not returned by APIs.
- Connector execution logs redact obvious `password`, `token`, and `secret` fields.
- HTTP connector URL policy blocks obvious private and metadata networks by default.
- Arbitrary SQL execution is not exposed.

## Production Hardening Checklist

- Replace development header authentication with real SSO or identity provider integration.
- Add authorization checks for model, task, datasource, connector, and ops actions.
- Replace local AES secret configuration with KMS, Vault, or another managed secret backend.
- Configure connector egress allowlists and network-level controls.
- Review audit retention, export, and tamper-resistance requirements.
- Configure TLS for frontend, backend, database, Redis, and object storage.
- Disable permissive development defaults for tenant and user identity.

## Data Separation

Koravo keeps process execution data in Flowable tables and platform/business governance data in `ko_*` tables. Large business payloads should be stored in domain tables or object storage and referenced by process variables, not embedded directly into Flowable variables.

## Known Limits

- Fine-grained RBAC is not implemented yet.
- Development header identity must not be used directly in production.
- Connector policy is a defensive default, not a complete SSRF solution by itself.
